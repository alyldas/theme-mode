# Contributing

## Local Checks

Inside a package checkout, use:

```sh
npm install
npm run check
```

## Commits

Use Conventional Commits for project history:

- `feat:` for user-facing additions.
- `fix:` for bug fixes.
- `docs:` for documentation-only changes.
- `test:` for test-only changes.
- `chore:` for tooling and maintenance changes.

Release preparation is handled by Release Please through release pull requests. Do not update package versions, `CHANGELOG.md`, or release tags manually. Package publishing runs from the Release Please workflow after a release is created.

Commit messages are checked locally by the Husky `commit-msg` hook through commitlint.

## Generated Files

Keep the repository source-only. These paths are build artifacts and must not be committed:

- `dist`
- `coverage`
- `.typecheck`
- `node_modules`
- `*.tgz`

`dist` is still required for git installs, packaging checks, and future npm publishing. It is generated by `npm run build`, `npm run test:exports`, `npm run pack:dry`, and `npm run prepare`.

README screenshot PNGs are committed documentation assets. Regenerate them from [examples/demo/index.html](examples/demo/index.html) when the demo UI changes.

## Scope

Keep v1 focused on:

- core helpers
- `useThemeMode`
- renderless `ThemeModeToggle`
- Nuxt module anti-flash integration
- optional preset CSS

Do not add storage-key migrations, a design system, arbitrary theme registry, or extra framework adapters without a separate proposal.

## Docs

Markdown links must be relative, so the package README works both on npm and GitHub.
